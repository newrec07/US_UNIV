# Initial Design Log — Admissions OS 설계 상세 기록

> 이 문서는 **대화의 맥락 전체를 재구성**하기 위한 것이다.
> `CLAUDE.md`가 "무엇을 결정했나(결론)"라면, 이 문서는 **"어떻게·왜 그 결정에
> 도달했나(과정)"** 를 담는다. 새 채팅이 원래 대화 기록에 접근하지 못해도,
> 이 로그를 읽으면 설계 의도와 거부한 대안까지 이해할 수 있다.
>
> 읽는 순서: `CLAUDE.md`(원칙·결정) → 이 문서(상세 근거) → `student-schema.ts`(결과물).

---

## 0. 작업의 전체 흐름 (어떻게 진행됐나)

이 스키마는 한 번에 그린 게 아니라, **블록을 하나씩 단계적으로 쌓고 → 두 번의
전체 점검으로 다듬는** 방식으로 만들어졌다. 사용자의 핵심 요구는 두 가지였다:

1. 단계별로 하나씩 세세하게(한 번에 쏟아내지 말 것).
2. 데이터 구조를 지금 견고히 잡아 나중의 유지비용 폭발을 막을 것.

진행 순서:
- **1단계**: Student 엔티티를 8개 논리 블록으로 나눠 하나씩 설계(§1).
- **2단계**: 두 방법론으로 전체 점검 — 행동번역(§2) + 구조내구성(§3).
- **3단계**: 점검에서 나온 횡단 발견 6개를 통합 처리(§4).
- **4단계**: 전체를 한 파일로 통합, 타입검증, 문서화(결과물).

## 1. 블록별 설계 — 무엇을 채택하고 무엇을 거부했나

### 블록1 — Identity & Context
- 처음엔 필드마다 긴 설명을 달았으나, 사용자가 "데이터형 간단·빠르게"를 원해
  필드명+타입+(필수/선택) 수준으로 낮춤.
- **학년 표현**: NZ Year 11 같은 로컬 학년을 미국 표준(GRADE_10)으로 변환 저장.
  사용자 요청으로 **원본 로컬 학년도 함께 보관**(안전). → 변환 테이블 필요.
- **타임라인 기준 = 졸업연도**로 합의(학년은 매년 바뀌지만 졸업연도는 고정).
- **권한/뷰 분리 제거**: 사용자가 "부모가 따로 볼 정보 없음, 모두 동일 오픈"
  결정 → System Meta가 가벼워짐.
- 인구통계(인종·1세대·사회경제)는 나중에 추가됨(블록6 인종 논의에서 파생).
  학생 속성은 블록1, 대학 분포는 블록6/College로 역할 분리.

### 블록2 — Academic Direction
- 핵심 통찰: **pre-med는 전공이 아니라 진로 트랙.** "전공 후보(intendedMajors)"와
  "진로 트랙(careerTrack)"을 분리.
- **fieldClusters가 진짜 기준점**: 전공명은 학생마다 제각각이라 시스템이 다루기
  어렵지만, 큰 계열(STEM 생명과학)은 안정적 → 대학리스트·활동추천·진단의 연결고리.
- **거부한 대안**: 처음 enum을 완전 고정으로 짰으나 사용자가 "너무 정형적,
  유연성 필요"라고 지적. 완전 자유 태그도 고려했으나, 사용자가 **자동완성 방식**을
  선택("그래야 agent가 수집·정리·분석 가능"). → `(string & {})` 패턴 채택:
  표준값 자동완성 추천 + 자유값 허용. 정형성(분석)과 유연성(현실) 양립.

### 블록3 — School Academics
- **핵심 경계 정리**(사용자 혼동 해소): 블록3=학교 성적표(transcript),
  블록4=시험 성적증명(score reports). 미국 대학이 실제로 둘을 따로 받음
  (성적표는 학교가, AP/SAT는 College Board가 전송).
- **AP의 이중성**: 학교에서 AP *수업*을 들으면 블록3 Course, AP *시험*만 따로
  보면 블록4. NZ 학생은 AP수업 없이 시험만 → 블록4.
- **원본/환산 분리**: reportedGpa(원본)와 estimatedUsGpa(환산)를 절대 안 섞음.
  환산은 근사치라 `gpaConversionConfidence`(신뢰도) 동반 → 환각 방지.

### 블록4 — Standardized Tests
- **2단 구조**: TestTrack(시험 종류) + attempts[](응시 기록). superscore·점수
  추이 추적 위해. SAT는 보통 2~3번 봄.
- **isOfficial**: 공식 vs 모의/진단 분리. "SAT diagnostic"은 계획용, 공식은 제출용.
- **status로 "안 본 시험"도 1급 데이터**: "No SAT plan yet" 리스크가
  `NOT_STARTED`로 표현. 안 한 것도 데이터.
- **TestType 보강**(웹검색으로 확인): CLT(325+ 대학, FL·UNC 인정), IB_EXAM
  (Yale test-flexible로 SAT 대체 가능), PSAT/PreACT, PTE/Cambridge 추가.
  사용자 지시로 검색에서 나온 것 전부 추가.

### 블록5 — Activities & Leadership
- **3분할**: activities / competitions / awards. Common App 구조(Activities 섹션 +
  Honors 섹션) 일치. 대회는 양면성(참가+수상)이라 `isAwardWorthy`로 처리.
- 사용자가 깜빡한 것 추가: **공식 대회·경진대회·에세이 컨테스트**(크림슨식).
- **가장 중요한 전환** — 사용자 위화감: "너무 결과 위주. 입상 못 하면 필드가 0이
  되는 게 안 좋다." → 결과 칸은 유지하되 **과정·방향 칸을 동등하게 추가**:
  - `growthStage`(EXPLORING~INITIATING): 입상과 **독립**. 입상 못 해도 단계로 진전 표현. 0이 아니라 단계.
  - `motivation`/`learnings`: "왜·무엇을 느꼈나" → **에세이 씨앗**(블록8 모듈로 흐름).
  - `nextSteps`의 `'suggested'` 상태: 빈칸을 비난이 아니라 "다음에 이걸 해봐"
    가이드로. → 나중에 통합 Action으로 흡수됨.
- 사용자 확인: "이런 게 있어야 추후 에세이 재료가 됨." → 데이터가 대시보드(행동)와
  에세이(이야기) 두 방향으로 흐르는 구조 확정.

### 블록6 — Financial & Preferences (가장 큰 블록)
- 사용자가 이 블록의 중요성을 강하게 지적: "추론 안 되는 것 빼고 모두 데이터화.
  세금신고부터 모든 재정을 넣어 현 상태 확인." 실화 근거: 지인 딸이 성적상
  UT Austin도 안 됐는데 노스이스턴 **전액**으로 감. 돈이 모자라 원서를 적게 쓴
  실수를 막아야 함.
- **두 엔진으로 분리**(웹검색으로 SAI 공식 확인):
  - 엔진 A(가족 재정 정량화): 세금신고 기반 SAI 입력값 → estimatedSAI 계산.
  - 엔진 B(대학별 재정 매칭): merit aggressiveness, gapping, net price로
    노스이스턴 케이스를 잡음. (나중에 College 정규화로 StudentCollege에 이동)
- **인종/인구통계 추가**(사용자 요청, "인종차별 아닌 데이터 기반"):
  웹검색으로 **SFFA 2023 판결** 확인 → 인종 기반 입학 위헌. **"쿼터" 개념은
  스키마에 넣지 않음**(법적으로 부재, Bakke 1978). 대신 공개 CDS 등록자 분포를
  "참고"로, 학생 인구통계는 자기보고·선택. 차별적 단정 방지 가드레일.

### 블록7 — Profile Assessment
- **핵심 결정**: 학생 입력이 아니라 **에이전트가 다른 블록을 읽고 생성하는 진단
  저장소.** 강약점은 이미 성적·시험·활동·재정에 데이터로 존재 → 중복 입력 불필요.
- **사실(facts) 블록과 다른 성격 = 판단(assessments) 블록** → 반드시 근거·시점·
  신뢰도 동반. `evidence[]` 필수(근거 없는 단정 = 환각).
- `critical_gap`을 비난 아닌 가이드로(블록5 교훈 계승).
- `nextBestAction` = 시스템의 심장. 수많은 약점 중 "지금 단 하나".
- 사용자 질문("consultant가 LLM인가?")으로 명확화: agent=LLM, consultant=사람.
  사용자 선택: **LLM 기본, consultant 값은 나중 위해 남김.** modelUsed로 어느 LLM
  인지도 기록(Claude/GPT 병행 디버깅용).

### 블록8 — System Meta
- 권한 분리를 뺐으므로 가벼움. completeness(에이전트가 빈 곳 찾는 지도),
  schemaVersion(마이그레이션 대비), dataSources(출처 추적).

## 2. 첫 번째 점검 — 행동 번역 (Action-Translation Audit)

**왜**: 사용자 명제 — "달성률은 감독자 언어, 학생에겐 스트레스. 필요한 건 쉽고
명확하게 분리된 행동 가이드." 우리가 만든 측정값(rating, completeness, risk…)이
점수판으로 변질될 위험 점검.

**5개 렌즈**: ①측정→행동 변환 ②인지부하(1~3개) ③원자성(30분) ④빈 곳=초대
⑤자동 우선순위. **처분**: (A)유지 / (B)번역레이어 추가 / (C)학생 첫화면에서 내림.

**블록별 핵심 처분**:
- 블록4(시험): NOT_STARTED→"SAT 날짜 잡기"로 자연스럽게 번역. 우등생.
- 블록5(활동): ActivityStep의 'suggested'는 훌륭. 단 growthStage/depth 등급은
  학생에게 점수판이 되니 첫화면에서 내림(근거로만 내부 사용).
- 블록6(재정): affordabilityVerdict 등은 부모 의사결정 데이터. 학생 일일 행동 아님.
- 블록7(진단): 가장 위험. rating·risk는 status_layer로, 학생 첫화면엔
  nextBestAction+행동만.

**산출물 3가지**:
- 추가1: **통합 Action 엔티티**(흩어진 행동을 한 큐로 → 우선순위 가능).
- 추가2: ~~audience(숨김 플래그)~~ → **defaultSurface(노출 깊이)로 수정**.
- 추가3: nextBestAction을 시스템 단일 출력으로 승격.

### ★ 거부하고 수정한 것 (중요)
처음엔 추가2를 `audience: student/parent/internal_only`(접근 **차단**)로 설계.
사용자 반론: "학생이 자기 현 상황 아는 것도 나쁘지 않다." → **수정**:
숨김이 아니라 **`defaultSurface`(기본 노출 깊이)**. action_layer(첫화면) /
status_layer(원할 때 보는 현황) / system_layer. **모두 학생에게 열림. 들이밀지
않을 뿐.** 근거 못 보면 행동이 블랙박스 명령이 됨(블록7 evidence 철학과도 일치).
스트레스 원인은 정보의 존재가 아니라 원치 않을 때 강제 노출.

## 3. 두 번째 점검 — 구조 내구성 (Structural Durability Audit)

**왜**: 사용자 요구 — 나중에 필드 땜질하면 유지비용 폭발. 지금 견고히.
**함정 경계**: "모든 걸 미리 넣기"(over-engineering)가 아니라 "나중에 추가해도
안 깨지는 구조". 변경 종류: (가)값 추가=안전(이미 `(string & {})`로 대비) /
(나)구조 변경=위험. **주로 (나)를 사냥.**

**6개 렌즈**: ①enum 개폐 ②단수/복수 ③평면/중첩 ④시간축(스냅샷/이력)
⑤출처·신뢰 ⑥관계·참조(이름/ID). **처분**: (A)견고/(B)지금 유연화/(C)의도적 보류.

**블록별 핵심 발견**:
- **블록1**: "학교"가 렌즈2·3·4·6에 동시에 걸림 → 단수 문자열을 `School[]`
  객체·이력·id로 승격(전학 대응). gradeStandard는 저장→계산 전환.
  residencyStatus 열기(DACA·난민 등). 
- **블록2**: intendedMajors를 가벼운 객체(MajorInterest)로. **fieldClusters가
  시스템 공유 어휘**임을 발견(통합 점검으로 이관).
- **블록3**: 성적을 `School→Term→Course` 3층 계층으로. GpaValue 객체화 +
  **가중/비가중 GPA 누락 보강**(중요 — 미국 입시 필수인데 빠져 있었음).
- **블록4**: 우등생(attempts[] 이력 이미 견고). 점수를 ScoreValue로 구조화 +
  만점(maxValue) 맥락 추가.
- **블록5**: growthStage를 스냅샷→선택적 마일스톤 이력으로. CompetitionResult
  객체화("500명 중 3등" 표현). Award-Competition을 **이름→ID 연결**(중복 해소).
- **블록6**(발견 최다): 부모재정 8필드를 Household 객체로(이혼가정 noncustodial
  대응). **net price를 grant/merit/loan 분해**(노스이스턴 케이스 정밀화).
  collegeId 필수화. FinancialFigure에 출처.
- **블록7**: **진단을 이력화**(current+history). trend의 모순 해소 — trend 필드는
  있는데 계산할 과거가 없던 문제를, 이력 쌓아 계산으로. evidence 구조화.
- **블록8**: dataVersion을 schemaVersion과 분리. completeness 세분화.
  출처 어휘 통일 필요 발견.

### 반복된 패턴 (세 번 나타남)
"추세를 보려면 스냅샷이 아니라 이력이 필요" — 블록3(GPA추세)·5(성장궤적)·
7(진단trend)에서 동일하게 발견 → 시스템 일관 원칙으로 승격.

### 누락 발견 (점검이 실제로 막은 것)
가중/비가중 GPA, net price grant/loan 분해, 이혼가정, 진단 trend 모순, 이름 기반
참조 — 전부 나중에 발견했으면 대규모 마이그레이션. 지금 반영해서 회피.

## 4. 통합 점검 — 횡단 발견 6개 처리

**왜**: 점검에서 블록을 넘나드는 발견들이 쌓임. 개별 블록이 아니라 시스템 구조로
해결. **순서 = 재료 → 원칙 → 계층 → 허브**(의존성 순. 재료가 다른 것의 기반).
각 발견마다 **소속·참조방식·블록수정** 3가지 결정.

### 재료1 — FieldCluster 공유 어휘
문제: 학생이 블록2에서 만든 자유 클러스터가 블록5 태깅에서 안 보임(비동기).
해결: 표준값=시스템 상수, 자유값=`student.vocabularies.customFieldClusters`에
등록→전 블록 재사용. 모든 블록 `FieldClusterRef`로 통일.

### 재료2 — College 공유 엔티티 ★ 큰 갈림길
문제: 한 대학에 객관 사실(COA·merit, 모든 학생 동일)과 학생 관계(net price,
학생마다 다름)가 섞여 있음. 지금 CollegeFinancialFit이 둘을 한 덩어리로 둠.
**사용자 결정**: 3가지 길(전부 Student 안 / 별도 공유 엔티티 / 하이브리드) 중
**길2(정규화)** 선택. → College를 Student 밖 공유 엔티티로. 객관 사실 한 벌,
학생은 `StudentCollege`로 collegeId 참조 + 관계만 보유. **이 결정으로 시스템이
"Student 한 덩어리"에서 "다중 최상위 엔티티"로 전환됨.** 부수효과: 안 만들었던
대학리스트 모듈 뼈대(StudentCollege[])가 생김. blame2에서 보류했던 appliedMajor도
여기 제자리 찾음.

### 재료3 — DataSourceType 출처 어휘 통일
문제: 블록3·4·6·8이 제각각 출처 enum(agent_estimated/estimate/inferred 등).
해결: 시스템 공통 `DataSourceType` 하나로. **출처와 신뢰를 두 축으로 분리**
(출처=어디서, 신뢰=얼마나, 독립). 같은 "문서 업로드"라도 스캔 흐리면 신뢰 다름.

### 원칙 — "추세는 이력에서 계산" 정립
판별 기준 3질문: (1)변하나 (2)궤적이 의미있나 (3)되돌아볼 일 있나 → 셋 다 yes면
이력. 공통 골격 `TimePoint<T>` + `Trend`(저장 아닌 계산). 강제 교체가 아니라
정합 지침. residencyStatus는 변하지만 궤적 무의미 → 스냅샷(기준과 일치).

### 계층 — School/Term + 블록1↔3 연결
한 학교가 신원(블록1 School)과 성적(블록3 AcademicRecord)으로 나뉘어 살되
schoolId로 연결. College 정규화와 같은 논리(한 실체의 다른 측면을 적절한 자리에).
**School은 지금 학생 안에 두되 미래 공유 승격 자리 확보**(id 보유). over-eng 없이
미래 차단.

### 허브 — 통합 Action ★ 두 방법론이 만남
Action은 철저히 학생별 → Student 안 최상위 큐(블록 밖). 블록5의 ActivityStep/
preparationSteps, 블록7의 recommendedActions/nextBestAction을 **전부 흡수**.
각 행동은 origin으로 출처를 ID 참조. nextBestAction은 저장 아닌 **큐에서 계산**.
→ 블록은 "무엇이 어떤가"(상태)만, 큐는 "그래서 뭘 할까"(행동) 전담.
**의미**: 첫 방법론(왜=행동을 모아 우선순위)과 두 번째(어떻게=흩어지면 구조 깨짐)이
같은 결론에 다른 이유로 도달 → 설계가 옳다는 강한 신호.

## 5. 최종 통합 후 상태

- 전체를 `student-schema.ts` 한 파일로 통합, `tsc --strict --noEmit` **검증 통과**
  (모든 ID 참조가 깨끗이 맞물림 = 흩어진 결정들이 구조적으로 일관).

파생물(Prisma·JSON Schema), 아키텍처 미결정(모델 분담·규칙엔진/LLM 경계).
→ 상세는 `CLAUDE.md` §5 참조.
