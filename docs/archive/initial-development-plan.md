# Initial Development Plan — Admissions OS

> 이 문서는 Admissions OS를 아이디어 모음에서 실행 가능한 제품으로 발전시키기 위한
> **개발 순서, 자동 작업 규칙, 검증 기준, 사용자 의사결정 지점**을 정의한다.
>
> 설계의 이유는 `CLAUDE.md`, 상세 논의는 `DESIGN_LOG.md`, 타입의 현재 기준은
> `student-schema.ts`에서 확인한다. 문서끼리 충돌하면 무조건 한쪽을 조용히 따르지 말고
> 아래의 “불일치 처리 절차”를 적용한다.

---

## 0. 개발 목표

Admissions OS의 목표는 미국 대학 입시 정보를 많이 보여주는 것이 아니다.

학생의 현재 데이터를 읽고:

1. 객관적 상태를 계산하고,
2. 근거를 갖춘 진단을 만들고,
3. 지금 할 수 있는 1~3개의 작은 행동으로 번역하고,
4. 행동 완료에 따라 다음 행동과 타임라인을 자동 갱신하는 것

이 핵심 흐름이다.

```text
학생·가족 입력 / 공식 문서 / 검증된 외부 데이터
                    ↓
          정규화된 도메인 데이터
                    ↓
     결정적 규칙 엔진(계산·일정·우선순위)
                    ↓
       근거 기반 AI 진단·설명·코칭
                    ↓
        학생별 단일 Action 큐 1~3개
                    ↓
        완료·변경 이벤트 → 다시 계산
```

LLM은 사실과 숫자를 만들어내는 계산기가 아니다. 규칙 엔진과 검증 데이터가
“무엇을” 결정하고, LLM은 근거를 인용해 “왜 이것이 중요한지”와 “어떻게 시작할지”를
학생의 언어로 설명한다.

### 시스템의 역할 분리

이 프로젝트는 Next.js 안에 화면·업무 로직·에이전트를 모두 넣는 단일 웹앱으로 만들지 않는다.
웹, API 서버, 에이전트 실행기를 명확히 분리한다.

```text
┌──────────────────────────────────────────────────────────┐
│ Web Client — Dashboard + Messenger                       │
│ Next.js                                                   │
│                                                          │
│ - 서버 상태를 조회해 화면에 표현                         │
│ - 사용자의 메시지와 명시적 의사결정을 서버에 전달        │
│ - Agent 응답과 데이터 변경 이벤트를 실시간으로 수신       │
│ - 학생 기록의 최종 원본을 브라우저에 두지 않음            │
└──────────────────────────┬───────────────────────────────┘
                           │ HTTPS: REST commands/queries
                           │ SSE: message tokens/status/data events
┌──────────────────────────▼───────────────────────────────┐
│ Application API Server — System of Record                │
│ NestJS                                                    │
│                                                          │
│ - 인증·권한·입력 검증                                    │
│ - 학생·대학·Action·타임라인 데이터의 유일한 변경 관문     │
│ - 사용자의 질문과 의사결정을 Conversation에 기록         │
│ - 업무 규칙과 transaction 실행                           │
│ - Agent 작업을 Queue에 등록                              │
│ - 변경 결과를 Web에 이벤트로 발행                        │
└───────────────┬──────────────────────┬───────────────────┘
                │                      │
       ┌────────▼────────┐    ┌────────▼──────────────────┐
       │ PostgreSQL       │    │ Redis + BullMQ            │
       │ 영속 데이터      │    │ Agent job·재시도·상태     │
       └─────────────────┘    └────────┬──────────────────┘
                                       │
                          ┌────────────▼───────────────────┐
                          │ Agent Worker / Orchestrator    │
                          │ NestJS standalone worker      │
                          │                                │
                          │ 1. 서버가 허용한 도구로 조회    │
                          │ 2. 규칙 엔진 결과 수집          │
                          │ 3. Claude API에 질문            │
                          │ 4. 도구 호출을 서버에서 실행    │
                          │ 5. 답변·제안·근거를 저장        │
                          │ 6. Messenger로 결과 스트리밍    │
                          └────────────────────────────────┘
```

핵심 경계:

- **웹은 화면과 사용자 상호작용을 담당하며 데이터의 기준이 아니다.**
- **API 서버가 모든 영속 데이터와 업무 규칙의 기준이다.**
- **Agent는 DB를 직접 읽거나 쓰지 않는다.** 서버가 공개한 타입 안전 도구를 호출한다.
- 사용자의 채팅 문장이 곧바로 데이터 변경이 되지 않는다. Agent가 변경안을 구조화하고,
  서버가 검증하며, 중요한 변경은 사용자의 명시적 확인 후 transaction으로 반영한다.
- 데이터가 변경되면 대시보드는 다시 계산된 서버 상태를 수신해 갱신한다.
- Claude API 키, 시스템 프롬프트, 도구 구현은 브라우저에 절대 포함하지 않는다.

### 메신저에서 서버 상태가 바뀌는 흐름

```text
사용자 질문 전송
  → API가 Message 저장
  → AgentRun 생성 및 Queue 등록
  → Worker가 학생 데이터·규칙 결과를 서버 도구로 조회
  → Claude가 답변 또는 구조화된 변경 제안 생성
  → 답변은 Messenger에 스트리밍
  → 변경 제안은 사용자에게 확인 요청
  → 사용자가 승인
  → API가 Command 검증 및 transaction 실행
  → dataVersion 증가, 파생값·Action 재계산
  → Web이 변경 이벤트를 받아 Dashboard 갱신
```

읽기 질문은 곧바로 답할 수 있지만 다음과 같은 변경은 사용자 확인을 요구한다.

- 학생의 공식 성적·시험점수·재정정보 수정
- 대학 리스트 추가·삭제 또는 지원 상태 변경
- 후속 일정에 영향을 주는 Action 완료·취소
- 에이전트 추론값을 사용자 확인 사실로 승격하는 변경

## 1. 문서와 코드의 우선순위

각 자료의 책임을 다음처럼 나눈다.

| 자료 | 책임 |
|---|---|
| `CLAUDE.md` | 제품 철학, 변경하면 안 되는 핵심 결정 |
| `DESIGN_LOG.md` | 결정이 나온 과정과 거부한 대안 |
| `student-schema.ts` | 현재 도메인 타입의 기준 |
| `computations.ts` | 결정적 계산 규칙의 초안 |
| `sample-colleges.ts` | 개발·테스트용 대학 데이터 |
| `HOW_TO_DEVELOP.md` | 개발 순서, 자동화 범위, 품질 기준 |
| 테스트 | 구현이 실제로 지켜야 하는 실행 계약 |

### 불일치 처리 절차

문서와 코드가 다르면 다음 순서로 처리한다.

1. 단순한 문서 지연인지, 설계 결정의 변경인지 구분한다.
2. 기존 결정의 이유를 `CLAUDE.md`와 `DESIGN_LOG.md`에서 확인한다.
3. 동작을 바꾸지 않는 문서 지연은 자동으로 문서를 최신화한다.
4. 데이터 구조·사용자 경험·외부 비용·안전에 영향을 주는 변경은 사용자 결정을 받는다.
5. 결정 후 문서, 타입, 구현, 테스트를 같은 작업에서 함께 갱신한다.
6. 변경 이유와 날짜를 결정 기록에 남긴다.

현재 확인된 불일치:

- `CLAUDE.md`는 일부 계산 로직과 타임라인을 미구현으로 기록하지만 타입과 계산 초안은 이미 존재한다.
- 시스템 루트 설명에는 `calendars`가 누락되어 있으나 `System` 타입에는 포함되어 있다.
- `GRADE_8`은 `GradeLevel`에 없는데 계산 코드에서 `as any`로 우회한다.
- `StudentCollege.admitChance`는 “합격·불합격 예측 기능 없음” 결정과 충돌하므로 제거 대상이다.
- `reach/target/safety`는 합격 예측처럼 오해될 수 있어 목적과 명칭을 Phase 0에서 재검토한다.

이 항목들은 프로젝트 골격을 만들기 전에 정리한다.

## 2. 자동 개발의 기본 루프

AI 개발 에이전트는 한 작업을 다음 루프로 수행한다.

### 2.1 작업 선택

우선순위는 다음 순서다.

1. 빌드·타입·테스트를 막는 문제
2. 잘못된 재정·일정·입시 판단을 만들 수 있는 문제
3. 핵심 사용자 흐름을 완성하는 작업
4. 데이터 출처와 추적성을 강화하는 작업
5. 개발 편의와 코드 정리
6. 장식적 UI와 비핵심 기능

한 번에 하나의 수직 기능을 완성한다. 예를 들어 “재정 모듈” 전체를 한 번에 만드는 대신
“가족 재정 입력 → 검증 → SAI 추정 결과 → 근거 표시 → 테스트”를 하나의 완료 단위로 잡는다.

### 2.2 구현 전 확인

모든 기능은 다음 질문에 답해야 한다.

- 이 기능은 학생이 다음 행동을 정하는 데 어떤 도움을 주는가?
- 결과가 규칙 엔진에서 나와야 하는가, LLM에서 나와야 하는가?
- 저장할 사실인가, 이력에서 계산할 파생값인가?
- 값만 추가하면 확장되는가, 구조 변경이 반복될 가능성이 있는가?
- 출처와 신뢰도를 어디에 기록하는가?
- 잘못됐을 때 학생·가족에게 어떤 피해가 생기는가?

답이 불명확하면 작은 탐색 작업이나 설계 메모를 먼저 만든다.

### 2.3 구현

- 타입을 먼저 정의하거나 갱신한다.
- 순수 계산은 부작용 없는 함수로 작성한다.
- 외부 데이터 접근은 계산 로직과 분리한다.
- LLM 입력에는 필요한 데이터와 출처만 전달한다.
- LLM 출력은 런타임 스키마로 검증한다.
- 새 행동은 각 블록 안이 아니라 학생의 단일 `Action[]` 큐에 넣는다.
- 학생 첫 화면에는 최대 3개 행동만 노출한다.

### 2.4 검증

최소 검증 순서:

1. 포맷 및 정적 분석
2. TypeScript strict 타입 검사
3. 단위 테스트
4. 참조 무결성 검사
5. 대표 학생 시나리오 테스트
6. 학생 화면의 행동 번역 점검
7. 재정·법률·일정처럼 변하는 외부 사실의 출처 확인

검증하지 못한 항목은 완료로 표현하지 않고, 무엇을 확인하지 못했는지 명시한다.

### 2.5 문서 동기화

구현이 기존 설명과 달라졌다면 같은 작업 안에서 문서를 갱신한다.

- 핵심 결정 변경 → `CLAUDE.md`
- 결정 이유와 대안 → `DESIGN_LOG.md`
- 개발 순서나 품질 기준 변경 → `HOW_TO_DEVELOP.md`
- 사용법과 실행법 → 향후 루트 `README.md`

## 3. AI가 자동으로 진행해도 되는 작업

다음 작업은 기존 원칙 안에서 가역적이고 범위가 명확하므로 별도 승인 없이 진행할 수 있다.

- 파일 구조 정리와 모듈 분리
- TypeScript strict 오류 수정
- `any`, 중복 타입, 죽은 코드 제거
- 테스트 작성과 테스트 데이터 생성
- 기존 타입에서 JSON Schema 생성
- 계산 함수를 순수 함수로 분리
- 문서와 코드의 명백한 상태 불일치 수정
- 출처 URL, 조회일, 데이터 연도 필드 추가
- 로그, 오류 처리, 입력 검증 추가
- 접근성, 반응형 레이아웃, 빈 상태 문구 개선
- 기존 기능을 바꾸지 않는 리팩터링
- 보안 취약점과 개인정보 노출 위험의 안전한 수정

자동 작업도 다음 조건을 만족해야 한다.

- 사용자의 기존 변경을 덮어쓰지 않는다.
- 중요한 타입 또는 API 변경에는 마이그레이션 경로를 제공한다.
- 데이터 삭제나 대규모 이동은 먼저 대상을 확인한다.
- 실행 가능한 테스트가 있으면 작업 후 반드시 실행한다.

## 4. 반드시 사용자 결정을 받아야 하는 작업

다음은 선택에 따라 제품의 비용, 위험 또는 방향이 크게 달라지므로 구현 전에 질문한다.

### 제품 범위

- 첫 출시 사용자가 가족 4명 전용인지, 향후 다중 가족 SaaS까지 고려할지
- 부모와 학생의 권한을 계속 동일하게 둘지
- 지원서 제출과 대학 포털 자동화까지 제품 범위에 포함할지
- 인간 컨설턴트 기능을 언제 도입할지

### 기술 선택

- 웹 프레임워크와 배포 플랫폼
- 데이터베이스와 ORM
- 인증 제공자
- 주 LLM 제공자와 보조 모델 사용 여부
- 파일 저장소와 문서 OCR 서비스

### 비용과 외부 서비스

- 유료 API 또는 월별 비용이 발생하는 서비스 도입
- 대학 데이터의 유료 라이선스 구매
- 이메일·문자·캘린더 같은 외부 전송 기능

### 개인정보와 안전

- 미성년자 데이터 보존 기간
- 원본 성적표·세금 문서 저장 여부
- 사용자 데이터의 LLM 전송 범위
- 분석·오류 추적 서비스로 전송할 데이터 범위
- 데이터 삭제와 내보내기 정책

### 판단 정책

- reach/target/safety 분류 기준
- 재정 추정 결과의 허용 오차와 면책 표시
- 행동 우선순위 가중치 변경
- 기존 D1~D12 결정과 충돌하는 구조 변경

질문할 때는 단순히 “무엇을 원합니까?”라고 묻지 않는다. 추천안 하나와 대안들의
비용·장단점·되돌리기 어려운 정도를 함께 제시한다.

## 5. 권장 개발 단계

### Phase 0 — 설계 기준선 정리

목표: 현재 파일을 모순 없는 개발 출발점으로 만든다.

- `CLAUDE.md`의 완료·미완료 상태를 실제 코드와 맞춘다.
- `System` 루트에 `calendars`가 포함된 결정을 문서에 반영한다.
- `GRADE_8 as any` 문제를 해결한다.
- `StudentCollege.admitChance`를 제거하고 관련 문서·샘플·테스트를 정리한다.
- `reach/target/safety`를 예측이 아닌 객관적 리스트 구성 지표로 대체할지 검토한다.
- `student-schema.ts`를 도메인별 파일로 나눌지 결정한다.
- 모든 파생값과 저장값을 다시 구분한다.
- 날짜·화폐·국가 코드의 표현 규칙을 정한다.
- 현재 샘플 대학 데이터의 출처를 필드별로 기록한다.

완료 기준:

- 문서와 타입의 핵심 구조가 일치한다.
- `any` 없이 strict 타입 검사를 통과한다.
- 알려진 설계 부채가 목록화되어 있다.

### Phase 1 — 실행 가능한 TypeScript 프로젝트

목표: 누구나 같은 명령으로 설치·검증할 수 있는 최소 프로젝트를 만든다.

- pnpm workspace 기반 monorepo 구성
- Next.js 웹, NestJS API, NestJS Agent Worker 앱 생성
- `package.json`, 공통 `tsconfig`, 포맷터, 린터 구성
- 도메인 타입 export/import 정리
- API 계약과 런타임 입력 검증 도입
- PostgreSQL + Prisma 연결
- Redis + BullMQ 기반 Agent job 큐 연결
- REST 명령·조회 API와 SSE 이벤트 채널 구성
- CI에서 타입 검사와 테스트 실행
- 루트 `README.md`에 설치·검증 명령 기록

확정 구조 초안:

```text
apps/
  web/                 # Next.js Dashboard + Messenger
  api/                 # NestJS 업무 API, 인증, 데이터 변경 관문
  agent-worker/        # Queue 소비, Claude orchestration, tool loop
packages/
  domain/              # 프레임워크 비종속 도메인 타입·규칙
  contracts/           # API DTO, 이벤트, Agent tool schema
  database/            # Prisma schema/client
  config/              # 공통 설정·환경변수 검증
  testing/             # fixture와 테스트 도우미
docs/
```

통신 원칙:

- 일반 조회와 명령: REST
- Agent 답변 토큰, 실행 상태, 대시보드 갱신 이벤트: SSE
- 긴 Agent 작업과 재시도: BullMQ
- 사용자 의사결정 반영: API command + DB transaction
- WebSocket은 v1 필수로 넣지 않는다. 양방향 presence, typing, 다중 사용자 동시 편집이
  실제로 필요해질 때 추가한다.

완료 기준:

- 새 환경에서 설치 후 한 명령으로 타입 검사와 테스트가 실행된다.
- 도메인 코드가 UI나 특정 데이터베이스에 종속되지 않는다.
- 웹을 새로고침해도 Messenger와 Dashboard 상태가 서버에서 복원된다.
- 브라우저 코드에는 LLM 비밀키와 Agent 도구 구현이 존재하지 않는다.
- API와 Worker를 각각 재시작해도 저장된 메시지와 대기 작업이 유실되지 않는다.

### Phase 2 — 규칙 엔진 신뢰성 확보

목표: 학생의 중요한 판단을 만드는 계산을 검증 가능한 코드로 만든다.

우선순위:

1. 학년 및 입시 사이클 계산
2. Action 우선순위
3. completeness
4. 타임라인 상태와 마감 임박 계산
5. GPA 참고 환산
6. SAI 및 net price 추정

필수 개선:

- 모든 함수에 경계값과 실패 테스트를 작성한다.
- 현재 날짜는 함수 인자로 주입해 테스트 가능하게 한다.
- 같은 점수의 Action에 안정적인 tie-break 규칙을 둔다.
- 지난 마감일을 무조건 최우선으로 올리는 정책을 재검토한다.
- 알 수 없는 GPA 체계를 원본 값 그대로 미국 GPA로 통과시키지 않는다.
- SAI는 해당 연도의 공식 표와 계산 버전을 사용한다.
- FAFSA와 CSS Profile 추정을 서로 다른 엔진으로 분리한다.
- 재정 결과에는 계산 연도, 공식 버전, 입력 출처, 신뢰도, 면책 문구를 포함한다.

완료 기준:

- 결과가 같은 입력에서 항상 동일하다.
- 계산 결과가 어떤 버전과 근거에서 나왔는지 추적 가능하다.
- 위험한 입력은 조용히 계산하지 않고 오류 또는 `unknown`을 반환한다.

### Phase 3 — 타임라인과 Action 큐

목표: “언제 무엇을 해야 하는지”가 자동으로 연결되게 한다.

- `MilestoneTemplate` 실제 데이터 작성
- 외부 공식 일정과 템플릿 결합
- `generateTimeline()` 구현
- 마일스톤 상태 자동 계산
- 마감 접근 시 Action 생성
- 완료·건너뛰기·분기 선택에 따른 후속 일정 갱신
- 중복 Action 방지용 idempotency 규칙
- 첫 화면 1~3개 선택 로직 구현

완료 기준:

- 졸업연도와 학생 조건만으로 기본 타임라인이 생성된다.
- 같은 입력으로 재생성해도 행동이 중복되지 않는다.
- 학생에게 보이는 행동이 30분 이내의 구체적 단위다.

### Phase 4 — 근거 기반 AI 진단

목표: LLM이 학생 데이터를 읽고 설명하되 사실을 발명하지 못하게 한다.

- 서버 측 Orchestrator와 Claude tool-use loop 구현
- Agent에게 직접 DB 접근 대신 읽기·제안 도구 제공
- `ProfileAssessment` 출력 스키마 검증
- 각 진단에 하나 이상의 `Evidence` 요구
- evidence의 `sourceRef` 실재 여부 검사
- 근거 없는 rating·risk 거부
- 진단 결과에서 Action 후보 생성
- 규칙 엔진이 Action 우선순위를 최종 결정
- 모델명, 프롬프트 버전, 생성 시점, 데이터 버전 기록
- 모든 Agent 실행을 `AgentRun`으로 기록하고 실패·재시도 상태 추적
- 변경 도구는 기본적으로 즉시 쓰기 대신 `ProposedCommand`를 생성
- 사용자 승인 후 API 서버가 command를 실행하고 변경 이벤트 발행

완료 기준:

- 모든 진단 문장에서 근거 데이터로 이동할 수 있다.
- 존재하지 않는 데이터 참조는 저장되지 않는다.
- 모델을 다시 실행해 표현이 달라도 핵심 규칙 결과는 변하지 않는다.
- Agent가 서버의 허용된 도구 밖에서 데이터를 읽거나 변경할 수 없다.
- 사용자의 승인과 실제 변경을 감사 로그에서 연결할 수 있다.

### Phase 5 — 대학 리스트와 재정 매칭

목표: 명성 중심이 아니라 객관적 입시 현황·학업 적합성·실부담을 함께 비교한다.

- 대학 데이터 수집·검증 파이프라인
- 전공·교육환경·선호 적합성
- 학생별 net price 시나리오
- merit와 need-based aid 분리
- reach/target/safety 정책
- 데이터가 낡았거나 부족할 때 `unknown` 처리

완료 기준:

- 대학의 객관 데이터와 학생별 판단이 분리되어 있다.
- 모든 변동 데이터에 출처, 기준 연도, 갱신일이 있다.
- 장학금과 대출이 같은 “지원금”으로 합쳐져 보이지 않는다.

### Phase 6 — 최소 웹앱

목표: 한 가족이 실제로 사용할 수 있는 최소 제품을 만든다.

최소 화면:

- 서버와 연결된 Messenger
- 오늘의 행동 1~3개
- 행동 상세와 “왜 지금 이것인가”
- 학생 프로필 입력·수정
- 타임라인
- 대학 리스트와 비용 비교
- 상태·진단의 상세 보기
- 데이터 출처와 신뢰도 보기

화면 계층:

- `action_layer`: 오늘 할 행동
- `status_layer`: 현재 상태와 근거
- `system_layer`: 출처, 계산 버전, 모델 실행 정보

완료 기준:

- 빈 데이터가 실패나 빨간 점수로 보이지 않고 시작 행동으로 연결된다.
- 학생은 모든 상태와 근거를 찾아볼 수 있다.
- 모바일에서도 핵심 행동을 수행할 수 있다.
- Messenger의 사용자 승인 결과가 서버 데이터에 반영되고 Dashboard가 갱신된다.
- Agent 실행 중·완료·실패 상태가 화면에 명확히 표시된다.

### Phase 7 — 보안·운영·배포

목표: 미성년자와 가족의 민감 데이터를 안전하게 운영한다.

- 최소 권한과 비밀정보 관리
- 저장·전송 암호화
- 원본 문서 접근 로그
- 데이터 내보내기·삭제
- 백업과 복구 테스트
- 오류 추적 시 개인정보 제거
- LLM 요청·응답의 민감정보 최소화
- 대학 및 일정 데이터 갱신 작업

완료 기준:

- 개인정보 흐름과 보존 정책이 문서화되어 있다.
- 사용자가 자신의 데이터를 내보내고 삭제할 수 있다.
- 운영 장애와 외부 API 실패 시 안전한 대체 동작이 있다.

- 재정·합격 관련 불확실성이 명확히 표시되는가?

## 7. 외부 데이터 규칙

대학 정책, 시험 일정, FAFSA/CSS 공식, 학비, 합격률은 변동 데이터다.

각 레코드에 가능한 한 다음을 기록한다.

```ts
interface ExternalFactProvenance {
  sourceUrl: string;
  sourceTitle?: string;
  publisher: string;
  retrievedAt: string;
  effectiveYear?: string;
  confidence: 'official' | 'secondary' | 'estimated';
}
```

규칙:

- 공식 기관과 대학 공식 사이트를 우선한다.
- 검색 결과 요약만 근거로 저장하지 않는다.
- 서로 다른 연도의 수치를 한 레코드에서 섞지 않는다.
- 추정값과 공식값을 같은 신뢰도로 표시하지 않는다.
- 갱신 주기가 지난 데이터는 화면에 경고하거나 판단에서 제외한다.
- 합격·불합격 확률이나 개인별 예측값은 출처 유무와 관계없이 생성하지 않는다.

## 8. 완료의 정의

기능은 코드가 작성됐다는 이유만으로 완료되지 않는다. 다음을 모두 만족해야 한다.

- 문서와 코드가 동기화되어 있다.
- 남은 제한과 불확실성을 명시했다.

## 10. 확정된 제품·아키텍처 결정

### 결정 A — v1 제품 범위: 확정

**가족 4명 전용의 비공개 웹앱으로 시작하고, 다중 가족 SaaS는 보류한다.**

이유:

- 현재 동일 권한 모델과 잘 맞는다.
- 인증·조직·결제·테넌트 분리보다 핵심 입시 기능을 먼저 검증할 수 있다.
- 도메인 계층을 UI 및 사용자 수와 분리하면 추후 확장할 수 있다.

### 결정 B — 서버 중심 기술 스택: 확정

프로젝트는 다음 TypeScript monorepo로 구성한다.

- Web: Next.js + TypeScript
- API: NestJS
- Agent Worker: NestJS standalone application
- Database: PostgreSQL + Prisma
- Queue: Redis + BullMQ
- Contract validation: Zod
- Unit/integration test: Vitest
- Browser E2E: Playwright
- LLM: Claude API를 서버 Worker에서 호출
- Workspace: pnpm workspaces
- Development: Claude Code를 주 개발 에이전트로 사용

프론트엔드는 Dashboard와 Messenger를 제공하지만 상태의 기준은 API 서버다.
Agent는 Queue에서 실행되며 서버 도구를 통해 데이터와 규칙 엔진을 사용한다.

### 결정 C — 민감 원본 문서 저장: 구현 시점까지 보류

세금 신고서와 성적표 원본을 저장하지 않기로 확정한 것이 아니다. 이 항목은
미성년자 신원정보와 가족 재정정보가 포함된 파일 저장소를 만들 경우 필요한
암호화, 접근 로그, 보존 기간, 삭제, 백업 정책의 범위가 크게 달라지기 때문에 제시했다.

현재 결정:

- 구조화된 학생·재정 데이터는 서버 DB에 저장한다.
- Messenger 대화와 사용자의 의사결정도 서버에 저장한다.
- 원본 파일 업로드 기능은 해당 기능을 실제 구현할 때 저장 목적과 보존 정책을 다시 결정한다.
- 원본을 저장하게 되면 일반 DB 필드와 분리된 private object storage를 사용한다.

### 결정 D — 합격·불합격 예측 기능 없음: 확정

Admissions OS는 합격 또는 불합격을 예측하지 않는다. 합격 확률이나 개인별 예측치를
기능으로 만들지 않는다.

대신 다음의 객관적 현황과 분석 결과를 제공한다.

- 학교가 공개한 지원·등록 데이터와 해당 데이터의 연도·출처
- 학생의 학업, 시험, 활동, 일정, 재정 데이터 현황
- 데이터에 근거한 강점·빈 곳·준비 상태
- 대학별 학업·선호·재정 조건의 비교
- 지금 할 수 있는 다음 행동

`reach/target/safety` 같은 표현이 필요하더라도 합격 판정이 아니라 대학 리스트의
균형과 불확실성을 설명하기 위한 보조 분류로만 사용하며, 구현 전 별도 정책을 정한다.
