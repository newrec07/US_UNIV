# 개발 workflow

## 기본 루프

1. Explore: 관련 코드와 필요한 문서만 읽는다.
2. Plan: 수직 기능의 범위, 위험, 검증 기준을 정한다.
3. Implement: 타입에서 시작해 구현과 테스트를 함께 작성한다.
4. Verify: typecheck, 관련 테스트, 참조 무결성, 사용자 흐름을 확인한다.
5. Document: roadmap과 실제 변경된 기준 문서를 갱신한다.

작고 명확한 수정에는 긴 계획을 강제하지 않는다. 여러 앱·패키지·데이터 구조에 영향을 주는
작업은 구현 전에 계획과 사용자 결정을 확인한다.

## 작업 우선순위

1. 빌드·타입·테스트를 막는 문제
2. 잘못된 재정·일정·입시 판단을 만들 수 있는 문제
3. 핵심 사용자 흐름
4. 출처와 추적성
5. 개발 편의와 리팩터링
6. 장식적 UI

## 자동으로 진행 가능한 작업

- strict 타입 오류 수정
- 테스트와 fixture 작성
- 기존 기능을 바꾸지 않는 리팩터링
- 문서와 코드의 명백한 상태 불일치 수정
- 입력 검증, 오류 처리, 로그 개선
- 출처 URL, 조회일, 데이터 연도 추가
- 개인정보 노출 위험의 안전한 수정

## 사용자 결정을 먼저 받을 작업

- 기존 제품 원칙 또는 ADR과 충돌하는 변경
- 유료 서비스, 배포 플랫폼, 외부 메시지 전송 도입
- 권한 모델과 사용자 범위 변경
- 민감 원본 문서 저장과 보존 정책
- 재정 추정의 사용자 표시 정책
- 대학 목록 분류 정책
- 데이터 삭제 또는 되돌리기 어려운 migration

질문할 때 추천안과 대안의 비용, 장단점, 되돌리기 어려운 정도를 함께 설명한다.

## 외부 공식 자료의 수치를 코드에 반영할 때

학생 입력이나 기존 DB 값처럼 수집되는 데이터는 타입과 범위로 검증이 쉽다. 하지만 법정 공식,
연도별 변수표, 정책 임계값처럼 외부 공식 문서에서 그대로 가져와야 하는 수치는 다르다 — 한 번
잘못 옮기면 그 오류가 계산 전체에 퍼지고, 사람의 1회성 확인 없이는 정확도를 보장할 수 없다.
자동 파싱(WebFetch 등)도 PDF 표처럼 복잡한 레이아웃에서는 신뢰할 수 없다. 이런 값은 다음
절차를 따른다.

1. Claude가 공식 출처(정부·기관이 발표한 문서, 가능하면 PDF/원문)를 찾는다.
2. Claude의 자동 도구로 표를 정확히 추출할 수 없으면 추측해서 채우지 않는다. 대신 문서 경로와,
   사용자가 로컬 분석 도구(오프라인 LLM 등)에 그대로 붙여 넣을 수 있는 추출용 프롬프트를 함께
   제시한다. 프롬프트에는 표 이름, 필요한 정확한 필드, "추정하지 말고 원문을 그대로 인용할 것"을
   명시한다.
3. 사용자가 로컬에서 그 프롬프트를 실행하고 결과를 붙여준다.
4. Claude는 그 결과를 추가 가공 없이 코드와 ADR에 반영한다 — 출처 제목, 버전, 발표일, 페이지를
   주석과 ADR에 함께 남긴다.
5. 도메인 모델에 없는 세부 입력 때문에 근사가 필요하면 조용히 덮지 않고 결과의
   `simplifications`/경고 필드 같은 곳에 명시적으로 남긴다.

이 절차는 결과에 크게 영향을 주는 법정 공식·연도별 변수표·정책 임계값에 우선 적용한다.
(사례: [FAFSA SAI Formula A 2025-26 표](../decisions/ADR-006-financial-estimate-display-policy.md))

## 문서와 코드가 충돌할 때

1. 문서 지연인지 설계 변경인지 구분한다.
2. 관련 ADR과 초기 설계 기록에서 이유를 확인한다.
3. 동작을 바꾸지 않는 문서 지연은 함께 수정한다.
4. 구조·제품·안전에 영향을 주는 변경은 사용자 결정을 받는다.
5. 결정 후 타입, 구현, 테스트, 문서를 같은 작업에서 갱신한다.

## 작업 보고

- 무엇이 달라졌는가
- 어떤 파일을 변경했는가
- 무엇으로 검증했는가
- 검증하지 못한 것은 무엇인가
- 다음 사용자 결정이 필요한가

샘플, mock, experimental 계산을 완료된 운영 기능처럼 표현하지 않는다.
